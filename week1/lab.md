## 4. ทฤษฎี/แนวคิดประกอบ LAB (สรุปสั้น)

### 4.1 System Context Diagram คืออะไร?

* เป็นแผนภาพที่แสดง:

  * **ระบบของเรา (Our System)** อยู่ตรงกลาง
  * **ผู้ใช้ / ระบบภายนอก** อยู่รอบ ๆ
  * **ความสัมพันธ์ / การสื่อสาร** ระหว่างระบบของเรา กับสิ่งภายนอก
* จุดประสงค์:

  * ทำให้ทีมเข้าใจ “ขอบเขต” ของระบบ
  * รู้ว่า **ใคร** ใช้ระบบ **อย่างไร** และ **มีระบบอื่นอะไรเชื่อมต่อบ้าง**

---

### 4.2 องค์ประกอบหลักของ System Context Diagram

1. **ระบบของเรา (Our System / Target System)**

   * เช่น “Shopee Platform”, “LINE Chat System”, “YouTube Service”
2. **Actors (ผู้ใช้/ผู้มีปฏิสัมพันธ์โดยตรง)**

   * เช่น End User, Admin, Seller, Rider, Content Creator ฯลฯ
3. **External Systems / Services**

   * เช่น Payment Gateway, Notification Service, Identity Provider (Google Login ฯลฯ)
4. **Flows / Interactions (ลูกศร)**

   * แสดงทิศทางการเรียกใช้หรือแลกเปลี่ยนข้อมูล
   * เช่น `User → Our System` (ส่งคำขอ), `Our System → Payment Gateway` (ขอชำระเงิน)

---

### 4.3 ตัวอย่าง ASCII System Context Diagram

```text
                  +------------------------+
                  |      End User          |
                  |  (Customer / Viewer)   |
                  +-----------+------------+
                              |
                              v
+-----------------------------+-----------------------------+
|                         Our System                        |
|                  (Example: Streaming Platform)            |
|                                                           |
|   +--------------------+    +--------------------------+  |
|   |  Web / Mobile App  |    |   Admin Console          |  |
|   +--------------------+    +--------------------------+  |
|                                                           |
+-----------------------------+-----------------------------+
          ^                          ^
          |                          |
          |                          |
 +--------+--------+       +--------+--------+
 | Payment Gateway |       |  Analytics /    |
 |  (3rd Party)    |       |  Recommender    |
 +-----------------+       +-----------------+
```

**ข้อสังเกต**

* กล่องกลางคือระบบหลักของเรา
* Actors / External Systems ห้อมล้อมรอบและมีลูกศรเชื่อม
* ไม่ต้องละเอียดถึงระดับฐานข้อมูล/ตาราง แค่ระดับ “ใครคุยกับใคร” ก็พอ

## 6. งานที่ต้องทำ (รายละเอียดขั้นตอน)

> **เวลารวม: ประมาณ 3 ชั่วโมง**

### ขั้นที่ 1 – แบ่งกลุ่มและเลือกระบบ (ประมาณ 20 นาที)

1. แบ่งกลุ่ม กลุ่มละ **3–5 คน**
2. เลือกระบบซอฟต์แวร์ 1 ระบบที่ทุกคนในกลุ่มรู้จักและเคยใช้จริง เช่น

   * Chat: LINE, Facebook Messenger
   * E-Commerce: Shopee, Lazada
   * Streaming: YouTube, Netflix, Spotify
   * Food Delivery: Grab, LINE MAN, Robinhood
3. กรอกข้อมูลในส่วนที่ 7 (แบบฟอร์มสำหรับนักศึกษา)

   * ชื่อกลุ่ม
   * สมาชิกในกลุ่ม
   * ระบบที่เลือกและเหตุผลสั้น ๆ

---

### ขั้นที่ 2 – วิเคราะห์ Actors และ External Systems (ประมาณ 30–40 นาที)

1. ระบุ **Actors** ที่สำคัญ

   * End User ประเภทต่าง ๆ (Customer, Seller, Creator ฯลฯ)
   * Admin หรือ Operator (ถ้ามี)
2. ระบุ **External Systems** ที่เชื่อมต่อ

   * Payment Gateway, Bank API
   * Google/Facebook Login
   * External Analytics, Notification Service ฯลฯ
3. บันทึกลงในตาราง “Actors & External Systems” ในส่วนที่ 7

---

### ขั้นที่ 3 – กำหนดขอบเขตของ “ระบบของเรา” (ประมาณ 20 นาที)

1. ตอบคำถามในกลุ่ม:

   * “เราจะถือว่าอะไรคือระบบหลักของเรา?”
   * เช่น สำหรับ Shopee:

     * “Shopee Platform” (รวม Web + Mobile + Backend)
   * สำหรับ LINE:

     * “LINE Chat System” หรือจะโฟกัสเฉพาะ “Messaging Core” ก็ได้
2. สรุปเป็นคำอธิบายสั้น ๆ (2–3 บรรทัด)

   * เขียนในหัวข้อ “คำอธิบายระบบของเรา” ในส่วนที่ 7

---

### ขั้นที่ 4 – วาด System Context Diagram (แบบร่าง) (ประมาณ 40–50 นาที)

1. ร่างแบบคร่าว ๆ บนกระดาษ/กระดาน:

   * วาดกรอบกลาง = ระบบของเรา
   * วาง Actors และ External Systems รอบ ๆ
   * วาดลูกศรแสดงทิศทาง และเขียน label สั้น ๆ เช่น

     * “Login Request”
     * “Place Order”
     * “Send Message”
     * “Process Payment”
2. ตรวจสอบความสมเหตุสมผล:

   * มี Actors ครบหรือยัง?
   * มีระบบภายนอกสำคัญตกหล่นไหม?
   * มีการสื่อสารหลัก ๆ แสดงในรูปครบหรือไม่?

---

### ขั้นที่ 5 – จัดทำแผนภาพในเครื่องมือ (draw.io / อื่น ๆ) (ประมาณ 40 นาที)

1. เลือกเครื่องมือที่กลุ่มจะใช้ (เช่น draw.io)
2. วาดแผนภาพ System Context ฉบับเนียน (clean) ตามแบบร่าง
3. ตรวจสอบให้มี:

   * ชื่อ Diagram (เช่น `System Context – Shopee Platform`)
   * ชื่อ Actors และ External Systems ชัดเจน
   * ลูกศรมีทิศทาง และ label ดีพอที่จะ “เดาไม่ผิด”
4. บันทึกไฟล์เป็น:

   * ภาพ (PNG/JPEG) 1 รูป
   * และไฟล์ต้นฉบับ (เช่น `.drawio`) ถ้าใช้ diagrams.net

---

### ขั้นที่ 6 – เตรียมสรุปผลและการนำเสนอ (ประมาณ 30 นาที)

1. กรอกส่วน “สรุปผลการวิเคราะห์ของกลุ่ม” ในเอกสาร
2. เตรียมประเด็นสำหรับนำเสนอหน้าชั้น (3–5 นาที)

   * ระบบอะไร? ทำอะไร? ใครใช้?
   * Actors และ External Systems ที่สำคัญคือใคร?
   * มี insight หรือจุดน่าสนใจอะไร (เช่น ถ้าผู้ใช้เพิ่มมาก ๆ จุดไหนน่าจะเป็นคอขวด)
3. ส่งมอบเอกสารและแผนภาพให้ผู้สอน (ตามช่องทางที่กำหนด)

## 8. เกณฑ์การประเมินผล LAB 1 (Rubric)

> ผู้สอนสามารถปรับคะแนนรวมได้ตามต้องการ
> ตัวอย่างนี้ใช้คะแนนรวม **10 คะแนน** ต่อกลุ่ม + การมีส่วนร่วมรายบุคคล (เช่น +0–2)

### 8.1 การประเมินงานรายกลุ่ม (10 คะแนน)

| ด้านที่ประเมิน                                        | คำอธิบาย                                                                                   | คะแนนเต็ม |
| ----------------------------------------------------- | ------------------------------------------------------------------------------------------ | --------- |
| 1. ความครบถ้วนของการวิเคราะห์บริบทระบบ                | ระบุนิยามระบบ, Actors, External Systems ได้ครบถ้วนเหมาะสม                                  | 3         |
| 2. ความถูกต้องและสมเหตุสมผลของ System Context Diagram | Diagram สื่อสารโครงสร้างและการเชื่อมต่อได้ถูกต้อง ทิศทางลูกศรเหมาะสม ไม่มีสิ่งที่ผิดชัดเจน | 3         |
| 3. ความชัดเจนในการนำเสนอและคำอธิบาย                   | สามารถอธิบาย diagram ได้เข้าใจง่าย มีโครงเรื่องในการนำเสนอ                                 | 2         |
| 4. ความเป็นระเบียบและคุณภาพของเอกสาร/รูปแบบ           | รูปแบบเอกสาร/แผนภาพอ่านง่าย จัดวางเป็นระบบ ใช้ชื่อสื่อความหมาย                             | 2         |
| **รวม**                                               |                                                                                            | **10**    |

ตัวอย่างการให้คะแนนแบบระดับคุณภาพ (ใช้ประกอบ):

* 90–100% ของคะแนนในหัวข้อนั้น → **ดีมาก** (เกือบไม่มีจุดบกพร่อง)
* 70–89% → **ดี** (มีข้อผิดพลาดเล็กน้อย)
* 50–69% → **พอใช้** (ยังมีช่องว่าง/ความไม่ชัดเจน)
* ต่ำกว่า 50% → **ต้องปรับปรุง**
