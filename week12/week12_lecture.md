## วัตถุประสงค์การสอน

### 1.1 Threat Modeling เบื้องต้น: STRIDE Framework และ DFD

**1.1.1** นักศึกษาสามารถอธิบายแนวคิด Threat Modeling และความสำคัญของการคิดเรื่อง Security ตั้งแต่ขั้นตอนการออกแบบสถาปัตยกรรม (Security by Design) ได้

**1.1.2** นักศึกษาสามารถวาด Data Flow Diagram (DFD) ระดับ High-Level ของระบบ Task Board แบบ Microservices และระบุ Trust Boundary ได้

**1.1.3** นักศึกษาสามารถใช้ STRIDE Framework ในการวิเคราะห์ภัยคุกคาม (Threat) หลักของระบบและเสนอแนวทาง Mitigation เบื้องต้นได้

### 1.2 Security Architecture Components

**1.2.1** นักศึกษาสามารถอธิบายบทบาทของ Auth Service, JWT Token, API Gateway และ WAF ในสถาปัตยกรรมที่คำนึงถึง Security ได้

**1.2.2** นักศึกษาสามารถออกแบบ Authentication/Authorization Flow สำหรับระบบ Microservices โดยใช้ JWT Token ได้

**1.2.3** นักศึกษาสามารถปรับปรุง C4 Container Diagram (C2) และ Component Diagram (C3) ของ Task Board ให้สะท้อนกลไก Security ที่เพิ่มขึ้นได้

### 1.3 Zero-Trust Architecture

**1.3.1** นักศึกษาสามารถอธิบายหลักการ Zero-Trust ("Never Trust, Always Verify") และเปรียบเทียบกับ Perimeter-Based Security แบบดั้งเดิมได้

**1.3.2** นักศึกษาสามารถระบุจุดที่ควรใช้หลักการ Zero-Trust ในสถาปัตยกรรมของ Task Board System ได้

## สรุป CLO ที่เกี่ยวข้อง

### 🎯 CLO3 (Knowledge - K)
**อธิบายคุณลักษณะคุณภาพของซอฟต์แวร์ (Quality Attributes)** เช่น Performance, Scalability, Availability, **Security**, Modifiability ได้อย่างเป็นระบบ

> **ความเชื่อมโยง:** สัปดาห์นี้เน้น Security เป็น Quality Attribute หลัก โดยเรียนรู้ว่า Security มีผลกระทบต่อ Quality Attributes อื่น ๆ อย่างไร เช่น Security vs Performance (การเพิ่ม Encryption/Token Validation ทำให้ Response Time สูงขึ้น) และ Security vs Usability (การบังคับ MFA อาจลดความสะดวกของผู้ใช้)

### 🎯 CLO5 (Cognitive - C)
**วิเคราะห์ความต้องการของระบบ** เพื่อระบุ Architectural Drivers, Quality Attribute Scenarios และ Constraints ที่มีผลต่อการออกแบบสถาปัตยกรรมได้

> **ความเชื่อมโยง:** การทำ Threat Modeling ด้วย STRIDE คือการวิเคราะห์ Security Requirements อย่างเป็นระบบ เพื่อระบุ Security Drivers ที่ส่งผลต่อการออกแบบสถาปัตยกรรม

### 🎯 CLO6 (Cognitive - C)
**ออกแบบสถาปัตยกรรมระดับสูงของระบบซอฟต์แวร์** โดยใช้ C4 Model และเลือกใช้ Architectural Styles/Patterns ที่เหมาะสมได้

> **ความเชื่อมโยง:** นักศึกษาจะปรับปรุง C2 (Container Diagram) และ C3 (Component Diagram) ของ Task Board System เพื่อเพิ่ม Security Components เช่น Auth Service, API Gateway, WAF และ Centralized Logging

### 🎯 CLO7 (Cognitive - C)
**บันทึกการตัดสินใจด้านสถาปัตยกรรม** (Architecture Decision Records - ADR) อย่างชัดเจนและมีเหตุผลรองรับได้

> **ความเชื่อมโยง:** การตัดสินใจเพิ่ม Security Components เข้าสู่สถาปัตยกรรมควรถูกบันทึกเป็น ADR ระบุ Context, Decision, Consequences และ Trade-offs

### 🎯 CLO10 (Practical - P)
**ทำงานเป็นทีมในการออกแบบและพัฒนาสถาปัตยกรรมซอฟต์แวร์** โดยแบ่งหน้าที่ สื่อสาร และบูรณาการงานร่วมกันได้

> **ความเชื่อมโยง:** กิจกรรม Threat Modeling เป็นงานกลุ่มที่ต้องอาศัยการระดมสมอง วิเคราะห์ร่วมกัน และแบ่งหน้าที่ในการวาด DFD / ระบุ Threats / เสนอ Mitigations

### 🎯 CLO14 (Analytical - A)
**ใช้เครื่องมือ DevOps พื้นฐาน** (เช่น Git, CI/CD, Monitoring) เพื่อเชื่อมโยงสถาปัตยกรรมกับการ Deployment ได้

> **ความเชื่อมโยง:** Security Tools เช่น WAF, Centralized Logging (ELK/Loki) และ Secret Management เป็นส่วนหนึ่งของ DevSecOps Pipeline ที่เชื่อมโยงกับสถาปัตยกรรมและการ Deployment

### 🔐 ทำไม Security ต้องคิดตั้งแต่ขั้นออกแบบสถาปัตยกรรม?

**Security by Design** คือหลักการที่ว่า ความปลอดภัยต้องถูกออกแบบเข้าไปในสถาปัตยกรรมตั้งแต่แรก ไม่ใช่ "แปะ" เข้าไปทีหลัง

**ตัวอย่างจากชีวิตจริง:**

ลองนึกถึงแอปที่นักศึกษาใช้ทุกวัน:

| แอปที่คุ้นเคย | Security ที่ซ่อนอยู่ | ถ้าไม่มี จะเกิดอะไร? |

### 📊 Threat Modeling คืออะไร?

**Threat Modeling** คือ กระบวนการวิเคราะห์ระบบอย่างเป็นระบบเพื่อหาว่า:
1. **อะไรที่อาจถูกโจมตี?** (Assets)
2. **ใครที่อาจโจมตี?** (Threat Actors)
3. **โจมตีอย่างไร?** (Threats)
4. **ป้องกันอย่างไร?** (Mitigations)

```
┌────────────────────────────────────────────────────────────────┐
│                  Threat Modeling Process                       │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  Step 1                Step 2               Step 3             │
│  ┌──────────┐        ┌───────────┐       ┌───────────┐         │
│  │  วาด DFD │  ───►  │  ระบุ      │  ───► │  จัดลำดับ   │         │
│  │  ของระบบ │        │  Threats  │       │  ความสำคัญ │         │
│  │          │        │  (STRIDE) │       │  (Risk)   │         │
│  └──────────┘        └───────────┘       └───────────┘         │
│       │                                        │               │
│       │               Step 4                   │               │
│       │              ┌───────────┐             │               │
│       └──────────►   │  เสนอ     │  ◄──────────┘               │
│                      │Mitigation │                             │
│                      │  & ปรับ    │                             │
│                      │  Arch     │                             │
│                      └───────────┘                             │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 🛡️ STRIDE Framework

**STRIDE** เป็น Framework ที่ Microsoft พัฒนาขึ้น ใช้สำหรับจำแนกประเภทภัยคุกคามอย่างเป็นระบบ โดยแต่ละตัวอักษรแทนภัยคุกคามประเภทหนึ่ง:

```
┌──────────────────────────────────────────────────────────────────────────────┐
│                         STRIDE Threat Categories                             │
├──────────┬──────────────────────┬────────────────────────────────────────────┤
│ ตัวอักษร   │ ภัยคุกคาม              │ คำอธิบาย                                    │
├──────────┼──────────────────────┼────────────────────────────────────────────┤
│          │                      │                                            │
│    S     │ Spoofing             │ ปลอมตัวเป็นคนอื่น                              │
│          │ (การปลอมตัว)          │ เช่น ใช้ credentials ของคนอื่น Login           │
│          │                      │                                            │
├──────────┼──────────────────────┼────────────────────────────────────────────┤
│          │                      │                                            │
│    T     │ Tampering            │ แก้ไขข้อมูลโดยไม่ได้รับอนุญาต                     │
│          │ (การแก้ไขข้อมูล)        │ เช่น แก้ไข Task ของคนอื่น, SQL Injection       │
│          │                      │                                            │
├──────────┼──────────────────────┼────────────────────────────────────────────┤
│          │                      │                                            │

### 🔑 Component 1: Auth Service (Authentication & Authorization)

**Authentication** = ยืนยันตัวตน ("คุณเป็นใคร?")

## 1.3 Zero-Trust Architecture: หลักการ "Never Trust, Always Verify"

### 🏰 จาก Castle-and-Moat สู่ Zero-Trust

**แนวคิดเดิม: Perimeter Security (Castle-and-Moat)**
เหมือนปราสาทที่มีคูเมือง — ถ้าเข้ามาในกำแพงได้ก็เชื่อถือหมด

**แนวคิดใหม่: Zero-Trust**
ไม่เชื่อใครเลย แม้อยู่ใน Network เดียวกัน ต้องตรวจสอบทุกครั้ง

```
┌──────────────────────────────────────────────────────────────────────┐
│         Perimeter Security vs Zero-Trust                             │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   ❌ Perimeter Security (Castle-and-Moat)                            │
│                                                                      │
│   ┌─────── Firewall (คูเมือง) ──────────────────────────────┐          │
│   │   ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐              │          │
│   │   │Svc A │──│Svc B │──│Svc C │──│ DB   │              │          │
│   │   └──────┘  └──────┘  └──────┘  └──────┘              │          │
│   │                                                       │          │
│   │   ภายในกำแพง → "เชื่อถือกันหมด" ← อันตราย!                 │          │
│   │   ถ้า Attacker เจาะเข้ามาได้ → เข้าถึงได้ทุกอย่าง             │          │
│   │                                                       │          │
│   │   🦠 Attacker เจาะ Svc A → เข้า Svc B, C, DB ได้หมด     │          │
│   └───────────────────────────────────────────────────────┘          │
│                                                                      │
│   ✅ Zero-Trust ("Never Trust, Always Verify")                       │
│                                                                      │
│   ┌──────────────────────────────────────────────────────┐           │
│   │   ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐             │           │
│   │   │Svc A │🔒│Svc B │🔒│Svc C │🔒│ DB   │             │           │
│   │   └──────┘  └──────┘  └──────┘  └──────┘             │           │
│   │      🔑        🔑        🔑        🔑                │           │
│   │                                                      │           │
│   │   ทุก Service ต้องพิสูจน์ตัวตน + ตรวจสิทธิ์ทุกครั้ง              │           │
│   │   แม้อยู่ใน Docker Network เดียวกัน!                      │           │
│   │                                                      │           │
│   │   🦠 Attacker เจาะ Svc A → ❌ เข้า Svc B ไม่ได้         │           │
│   │      เพราะ Svc B ต้องการ valid token + correct role   │           │
│   └──────────────────────────────────────────────────────┘           │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

### 📋 หลักการ Zero-Trust 5 ข้อ (High-Level)

```
┌───────────────────────────────────────────────────────────────────────────┐
│                    Zero-Trust Principles                                  │
├───────────────────────────────────────────────────────────────────────────┤
│                                                                           │
│  1. 🔒 Verify Explicitly (ตรวจสอบทุกครั้ง)                                   │
│     ├── ทุก Request ต้องมี Token                                             │
│     ├── ตรวจสอบทั้ง Identity + Authorization                                │
│     └── ตัวอย่าง: Task Service ตรวจ JWT ทุกครั้งที่เรียก API                      │
│                                                                           │
│  2. 🔑 Least Privilege Access (ให้สิทธิ์น้อยที่สุด)                               │
│     ├── User ได้สิทธิ์เฉพาะที่จำเป็น                                             │
│     ├── Service ก็ได้สิทธิ์เฉพาะที่จำเป็น                                         │ 
│     └── ตัวอย่าง: Task Service เข้าได้แค่ Task DB ไม่ได้เข้า Auth DB              │
│                                                                           │
│  3. 💥 Assume Breach (สมมติว่าถูกเจาะแล้ว)                                    │
│     ├── ออกแบบให้ถ้า 1 Service โดนเจาะ → ไม่กระทบตัวอื่น                        │
│     ├── Blast Radius ต้องเล็กที่สุด                                            │
│     └── ตัวอย่าง: แยก DB ของแต่ละ Service (Database-per-Service)             │
│                                                                           │
│  4. 📊 Monitor & Log Everything (บันทึกทุกอย่าง)                              │
│     ├── Centralized Logging ทุก Service                                    │
│     ├── Alert เมื่อพบ pattern ผิดปกติ                                         │
│     └── ตัวอย่าง: แจ้งเตือนเมื่อมี failed login > 5 ครั้งใน 5 นาที                  │
│                                                                           │
│  5. 🔄 Encrypt Everything (เข้ารหัสทุกอย่าง)                                  │
│     ├── Data in Transit → TLS/SSL (HTTPS)                                 │
│     ├── Data at Rest → Encrypted Database                                 │
│     └── ตัวอย่าง: Service-to-Service ก็ใช้ HTTPS ไม่ใช่ HTTP                    │
│                                                                           │
└───────────────────────────────────────────────────────────────────────────┘
```

## 📊 Security Trade-offs: สิ่งที่ต้องชั่งน้ำหนัก

เมื่อเพิ่ม Security เข้าไปในสถาปัตยกรรม ย่อมมี Trade-offs:

```
┌────────────────────────────────────────────────────────────────────────────┐
│                    Security Trade-offs                                     │
├────────────────────────────────────────────────────────────────────────────┤
│                                                                            │
│  Security ↑↑          vs          Performance ↓                            │
│  ┌────────────────────────────────────────────────────────────────┐        │
│  │  เพิ่ม JWT Validation ทุก Request  →  +5-20ms Latency             │        │
│  │  เพิ่ม WAF rules                  →  +2-10ms Latency             │        │
│  │  เพิ่ม Encryption (TLS)           →  +1-5ms Latency              │        │
│  │  เพิ่ม Audit Logging              →  +Disk I/O, Storage          │        │
│  └────────────────────────────────────────────────────────────────┘        │
│                                                                            │
│  Security ↑↑          vs          Usability ↓                              │
│  ┌────────────────────────────────────────────────────────────────┐        │
│  │  Strong Password Policy           →  ผู้ใช้จำรหัสผ่านยาก            │        │
│  │  MFA (2-Factor Authentication)    →  ขั้นตอน Login มากขึ้น         │        │
│  │  Token Expiry สั้น (15 นาที)        →  ต้อง Login บ่อย              │        │
│  │  CAPTCHA                          →  ผู้ใช้รำคาญ                  │        │
│  └────────────────────────────────────────────────────────────────┘        │
│                                                                            │
│  Security ↑↑          vs          Complexity ↑↑                            │
│  ┌────────────────────────────────────────────────────────────────┐        │
│  │  เพิ่ม Auth Service               →  +1 Container to manage      │        │
│  │  เพิ่ม API Gateway                →  +Configuration overhead     │        │
│  │  เพิ่ม WAF                        →  +Rule maintenance           │        │
│  │  Centralized Logging              →  +Infrastructure cost      │        │
│  └────────────────────────────────────────────────────────────────┘        │
│                                                                            │
│  🎯 หลักการ: Security ต้อง "เพียงพอ" ไม่ใช่ "สูงสุด"                              │
│     ต้องเลือกระดับที่เหมาะกับ Risk Level ของระบบ                                 │
│     Task Board ≠ Banking App → ไม่ต้อง Security เท่า Banking                  │
│                                                                            │
└────────────────────────────────────────────────────────────────────────────┘
```

│  4. Zero-Trust                                                              │

│     → แม้อยู่ใน Docker Network เดียวกันก็ต้องตรวจสอบ                               │
│                                                                             │
│  5. Trade-offs                                                              │
│     → Security vs Performance vs Usability vs Complexity                    │
│     → เลือกระดับที่เหมาะกับ Risk Level ของระบบ                                   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```
